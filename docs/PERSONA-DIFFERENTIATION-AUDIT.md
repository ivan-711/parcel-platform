# Persona Differentiation Audit

*Generated 2026-04-19 from a 20-agent parallel research pass (8 persona-journey + 8 surface + 4 cross-cutting). Read-only. No code modified.*

## Executive summary

Parcel collects an 8-persona signal at onboarding (`OnboardingPage.tsx:19-28`) but consumes it in only **two** non-trivial places downstream: the default-strategy selector on `AnalysisResultsPage.tsx:30-34, 126` and a beginner-only guidance banner at `AnalysisResultsPage.tsx:62, 371-385`. Persona never reaches `Dashboard.tsx`, `TodayPage.tsx`, `Properties*`, `Pipeline.tsx`, `Contacts*`, `PricingPage.tsx`, or any paywall component — the four highest-stakes conversion surfaces in the product. The single biggest miss is paywall messaging: every paywall reads from a generic `FEATURE_LABELS` map (`frontend/src/types/index.ts:110-119`), so a wholesaler whose business *requires* skip tracing sees the same upgrade pitch as a beginner who doesn't know what skip tracing is. **Top recommendation: ship persona-aware paywall copy first (S effort, ~15–25% projected upgrade lift on Pipeline/Skip Tracing/Mail Campaigns) and fix the indefensible `hybrid → buy_and_hold` default (the persona explicitly said "I do multiple," then we silently collapse it to one).**

## What's already persona-aware

Exhaustive list of every place persona shapes behavior today:

- **Onboarding collection** — `frontend/src/pages/OnboardingPage.tsx:19-28` (8 keys: `beginner, wholesale, flip, buy_and_hold, brrrr, creative_finance, hybrid, agent`); skip-button defaults to `hybrid` at line 85; persisted via `frontend/src/stores/onboardingStore.ts:9, 37-38`.
- **Persistence** — `backend/models/users.py:33` stores `onboarding_persona` as a string column.
- **Default strategy on AnalysisResultsPage** — `PERSONA_DEFAULT_STRATEGY` at `frontend/src/pages/analyze/AnalysisResultsPage.tsx:30-34`; consumed at line 126 to auto-select the first strategy tab.
- **Default strategy in ManualCalculator** — `PERSONA_TO_STRATEGY` at `frontend/src/pages/analyze/components/ManualCalculator.tsx:16-25`, consumed at line 121-122. Byte-identical to the AnalysisResultsPage map.
- **Beginner guidance banner** — `AnalysisResultsPage.tsx:62, 371-385` ("We started with Buy & Hold — the most common strategy for your first rental"). Dismissal tracked to localStorage and PostHog (`beginner_guidance_shown`, `beginner_guidance_dismissed`, lines 68-88).
- **Beginner button hides** — Pipeline button hidden for beginners at `AnalysisResultsPage.tsx:315-322`; Find Buyers button hidden at `AnalysisResultsPage.tsx:340-350`.
- **Sample deal seeding** — `backend/core/onboarding/sample_deals.py:229-397` defines one `SAMPLE_DEALS[persona]` per key (8 deals); `backend/routers/onboarding.py:32` calls `get_sample_deal(persona)` post-onboarding.
- **AI-narrative tone** — `PERSONA_EXPERIENCE_LEVEL` at `sample_deals.py:17-26` maps personas to `beginner/intermediate/experienced`; consumed at `backend/routers/analysis.py:297-308` to tune real-analysis narrative tone.
- **Beginner-specific sample narrative** — dedicated `_NARRATIVE_BUY_HOLD_BEGINNER` at `sample_deals.py:186-222` (defines terms inline, uses pedagogical voice); other personas share strategy-keyed narratives `_NARRATIVE_WHOLESALE` / `_FLIP` / `_BUY_HOLD` / `_CREATIVE` / `_BRRRR`.

That is the complete inventory. Nine touchpoints across two files of frontend logic plus two backend modules.

## What's collected but unused

Every other surface is persona-blind even though `useOnboardingStore` is one import away:

- **`Dashboard.tsx`** — imports `useOnboardingStore` at line 19, destructures at line 101, but **explicitly omits `persona`** from the destructure. The store is wired in but the field is ignored. Backend `dashboard.py` returns identical stats for every persona; personalization would have to be frontend-only.
- **`TodayPage.tsx`** — zero persona reads across `TodayPage.tsx`, `BriefingCards.tsx`, `PipelineSummary.tsx`, `TodayCashFlowChart.tsx`, `TaskList.tsx`, `NewUserView` (lines 196-251). The `NewUserView` empty state ("Welcome to Parcel" / "Analyze your first property…") is the first thing post-onboarding users see and it knows nothing about the persona they just chose ten seconds earlier.
- **Analyze flow inputs** — `AnalyzePage.tsx`, `AnalyzerFormPage.tsx`, `StrategySelectPage.tsx` (lines 17-43) show all 5 strategies in static order regardless of persona. A wholesaler who said "wholesale" still has to click through StrategySelectPage to pick wholesale.
- **`PropertiesListPage.tsx` / `PropertyDetailPage.tsx`** — zero persona reads. The list shows Cash Flow / MAO / Profit / Money-Left-In via a strategy switch (`PropertiesListPage.tsx:60-89`) but no persona-driven column ordering, sort default, or status filter.
- **`Pipeline.tsx` + `PaywallOverlay.tsx` + `FeatureGate.tsx` + `pipeline-empty.tsx` + `LockedFeaturePage.tsx`** — zero persona reads anywhere in the billing/paywall stack. Every gated feature shows the same generic copy from `FEATURE_LABELS` (`frontend/src/types/index.ts:110-119`).
- **`ContactsListPage.tsx` / `ContactDetailPage.tsx` / `ContactModal.tsx` / `useContacts.ts`** — zero persona reads. The `TYPE_OPTIONS` array at `ContactsListPage.tsx:28-36` is a fixed 8-type list (`seller, buyer, agent, lender, contractor, tenant, partner, other`) shown identically to all personas, even though a wholesaler never adds a tenant and a landlord never adds a JV partner.
- **`PricingPage.tsx`** — zero persona reads across the highest-stakes conversion surface in the app. Tier subtitles at lines 308, 380-381, 461 are generic; the Carbon `Most Popular` badge at lines 351-356 is hard-coded; the `CARBON_FEATURES` array (lines 35-47) is a fixed 11-item list with no persona reordering.
- **15+ shared empty states** — `frontend/src/components/EmptyState.tsx` is consumed by `Properties / Contacts / MailCampaigns / Financing / Obligations / SkipTracing / Rehabs / Buyers / Sequences / Portfolio / Transactions / Dispositions / CampaignAnalytics / PropertyDetail / ContactDetail` with no persona context. Several actively *contradict* the user's persona — e.g., Properties' "Add a property to add to your collection" assumes ownership for an `agent` persona who analyzes for clients.
- **Sample-deal narrative for `agent`** — `sample_deals.py:360-376` returns `_NARRATIVE_BUY_HOLD` (an investor-voice narrative talking about the user's personal cash flow). No `_NARRATIVE_AGENT` exists. The agent's first post-onboarding read tells them how *they* will earn 24.8¢ per dollar of *their* down payment — which inverts the agent mental model entirely.
- **Sample-deal narrative for `hybrid`** — `sample_deals.py:327-358` defines three scenarios (buy_and_hold, brrrr, wholesale) on one property, but only the first uses a polished `_NARRATIVE_BUY_HOLD` block. Scenarios 2 and 3 have one-sentence inline blurbs (lines 349, 355). Hybrid users see one fully fleshed strategy and two throwaway footnotes.

## Per-persona readiness summary

**`beginner`** — Best-served persona today. Has dedicated guidance banner, dedicated sample narrative (`_NARRATIVE_BUY_HOLD_BEGINNER`), and two button hides. Still missing: persona-aware metric subset on `KeyMetrics.tsx` (currently shows DSCR/cap-rate/CoC unfiltered), beginner-aware paywall framing, beginner-specific `NewUserView` copy, sample property is unrepresentative (60-yr duplex with 24.8% CoC sets unrealistic expectations — C2 §(e) Priority 2). Several findings already documented in `docs/UX-AUDIT-SIGNUP-TO-ANALYSIS.md` (P0 #2, P1 #4, P1 #6).

**`wholesale`** — Persona key is overloaded with the strategy key, which masks the gap: strategy-level branching exists in calculator/results but **persona-level** branching does not. Skip Tracing, Mail Campaigns, Buyers, Dispositions are wholesale's MUST-HAVE features (`SAD/personas/02-desiree-solo-wholesaler.md:73, 153, 260`) but treat all personas identically. Portfolio Card and CashFlow Chart on TodayPage are theater for wholesalers (no holdings to track).

**`flip`** — Backend infrastructure (flip.py, rehab tracker, financing instruments) is rich; frontend exposure is undifferentiated. Hard money / private money UI lacks flip-specific defaults (`AddInstrumentModal.tsx`); the multi-flip portfolio view Ray needs (`SAD/personas/03-ray-house-flipper.md:89`) doesn't exist. Currently sees BRRRR/buy_and_hold strategies in compare view despite never using them.

**`buy_and_hold`** — Strategy-level support is mature; persona-level support is the *dumping ground* for hybrid/agent/beginner defaults, which means downstream UI optimized for buy_and_hold is actually optimized for "we don't know." Missing: tenant/lease management, refinance modeling, portfolio cash-flow aggregation. Buyer/Disposition/Wholesaler tabs surface meaningless options.

**`brrrr`** — Persona surfaces only at narrative tone + default strategy. Capital-recycling dashboard, refinance-readiness checklist, ARV-validation surfaces — all the operating-OS features BRRRR investors need — don't exist. Backend calculator (`brrrr.py`) computes `capital_recycled_pct` and `infinite_return` correctly; frontend exposure is one tab among five.

**`creative_finance`** — Substantial backend infrastructure (sub-to/wrap/seller-finance instruments at `AddInstrumentModal.tsx:38-42`, Financing Dashboard with due-on-sale risk monitor, balloon timeline, wrap spread) but **none of it is persona-gated**. A beginner sees the same financing dashboard as a creative-finance specialist, defeating the differentiation. Per `RESEARCH/23-packaging-monetization-by-persona.md:295`, creative-finance monitoring is supposed to be a Pro-tier differentiator; in code it's universal.

**`hybrid`** — Worst-served. Default strategy maps to `buy_and_hold` (`AnalysisResultsPage.tsx:33`) which directly contradicts the persona's identity. `StrategyComparison` exists (line 504) but is hidden behind a tab. Skip-button on Onboarding also defaults to `hybrid` (`OnboardingPage.tsx:85`), so users who abandoned onboarding silently get the worst-served persona. Sample data correctly seeds 3 scenarios; UI shows 1.

**`agent`** — Worst content match. Receives `_NARRATIVE_BUY_HOLD` (investor voice talking about *your* cash flow), no client/lead model in Contacts, no branded PDF (despite a `brand_kit` column existing on `users.py`), no client-segmented portfolio. `PERSONA_DEFAULT_STRATEGY['agent'] = 'buy_and_hold'` is arbitrary — agents serve clients across all strategies.

## Contradictions surfaced during research

- **A1 (beginner) vs C3 (paywalls)**: A1 recommends hiding Pipeline/Find-Buyers from beginners (matches existing code). C3 §(b) recommends a *beginner-specific paywall pitch* on Pipeline ("learn the deal lifecycle"). Both are correct because they apply to different timeline stages — A1 is about results-page buttons (correctly hidden today), C3 is about the paywall a beginner would hit if they navigated to Pipeline anyway. Not a true contradiction; resolved by treating them as sequential.
- **A4 (buy_and_hold) wants wholesale CRM hidden vs A7 (hybrid) wants nothing hidden**: Not a contradiction — different personas. The implication is that any "hide for buy_and_hold" must be persona-gated, not strategy-gated, or it'll harm hybrid users.
- **C1 vs A4 on `beginner → buy_and_hold` default**: C1 calls it "defensible with caveats." A1 + audit doc P1 #4 say the beginner banner already explains it. Resolution: keep the default; deepen the guidance and add "compare other strategies" CTA.
- **C2 narrative count**: Original brief claimed 40 narratives (5×8); verified count is 6 dedicated + 1 beginner-specific = 7 polished narratives feeding 8 sample deals (hybrid reuses buy_and_hold; agent reuses buy_and_hold). The brief overestimated by 5.7×.
- **Sample data design vs frontend default for `hybrid`**: `sample_deals.py:327-358` *clearly* assumes hybrid is multi-strategy (3 scenarios on one property). `PERSONA_DEFAULT_STRATEGY['hybrid'] = 'buy_and_hold'` *clearly* assumes hybrid is single-strategy. The two files contradict each other; the frontend wins, which is the wrong answer.
- **Brief-listed paywalls that don't exist**: The original prompt listed 13 paywalls (Pipeline, Portfolio, Reports, Rehabs, Financing, Obligations, Transactions, Documents, Sequences, Skip Tracing, Mail Campaigns, D4D, Compliance). C3 confirms only 8 exist in code (`ai_chat, pdf_export, pipeline, portfolio, offer_letter, document_upload, skip_tracing, mail_campaigns`); D4D and Compliance are stub `LockedFeaturePage` routes; Rehabs/Financing/Obligations/Transactions are fully accessible (no paywall); Sequences uses `ComingSoonGate` (not a paywall). Recommendations should target the 8 real paywalls.

## Recommended next steps

1. **Ship persona-aware `PaywallOverlay` copy this sprint** (Opportunity #1). Smallest effort, highest projected conversion impact, fully reversible. Start with Pipeline + Skip Tracing + Mail Campaigns where C3's persona-relevance matrix is sharpest.
2. **Fix the `hybrid` default in the same PR** (Opportunity #2). Either route hybrid users to `StrategyComparison` as the landing view, or remove the default entirely and force a one-time "which strategy for this deal?" choice. The persona explicitly opted into multi-strategy thinking; we should honor that.
3. **Write `_NARRATIVE_AGENT` and improve hybrid scenarios 2/3** (Opportunity #5). Pure content, ~3 hours, no schema change. Fixes the most jarring mental-model mismatch (agents seeing investor-voice narratives).
4. **Wire persona into `Dashboard.tsx`** (Opportunity #7). The store is already imported; the destructure at line 101 simply omits `persona`. Adding it back unlocks every downstream Dashboard improvement and is a strong forcing-function for an experiment framework.
5. **Treat the `agent` persona as a research project, not a feature**. Per A8 + C1, agents need a fundamentally different mental model (clients-as-portfolio, branded reports, lower per-deal commitment) that doesn't slot cleanly into existing surfaces. Decide whether `agent` is a first-class segment with dedicated UX, or fold it into `hybrid` and remove the false promise on `OnboardingPage.tsx:27`. Ship the cleanup either way; current state collects the persona then ignores it, which is the worst possible answer.
